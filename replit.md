## Where things live

- Website UI: `artifacts/grace-high-school/src/App.tsx` (single-file sections; nav links live in two identical arrays) + `src/index.css` (responsive grids). Content sections read from the public GET APIs via generated hooks and fall back to in-file static arrays (`*_FALLBACK`) when the API is empty/unreachable. `src/main.tsx` provides the `QueryClientProvider` and wraps `<App/>` in a wouter `<Router base={BASE_URL}>`. The site is a single `App()` layout that always renders the chrome (utility bar, nav, footer, lightbox/video modals) and swaps only the middle: `isHome` (location==="/") renders the home sections; otherwise `renderDetailPage()` renders one of the three detail pages (see below).
- Admin dashboard: `artifacts/admin/` (slug `admin`, preview `/admin/`). `src/pages/Login.tsx` + `Dashboard.tsx`, `src/lib/auth.tsx` (session-backed auth context), `src/components/ContentSection.tsx` (generic CRUD/reorder/visibility panel; supports text/textarea/array/boolean fields + gallery image upload), `src/components/SiteTextSection.tsx` ("Section Text" tab — edits fixed section headings/paragraphs via GET/PATCH per key).
- API server: `artifacts/api-server/src/routes/` — `storage.ts` (presigned uploads + object serving), `resources.ts` (past papers / holiday work / application forms CRUD), `content.ts` (generic CRUD factory for the 8 content tables), `site-text.ts` (editable fixed section copy; canonical blocks defined in-route), `submissions.ts` (public POST + authed GET/PATCH/DELETE for enquiry/application form submissions; in-route per-IP rate limit + `website` honeypot), `auth.ts` (login/logout/me, scrypt + pg sessions), wired in `routes/index.ts`.
- API contract (source of truth): `lib/api-spec/openapi.yaml` → run codegen to regenerate hooks + zod schemas.
- DB schema: `lib/db/src/schema/` — `resources.ts` (downloadable resources), `content.ts` (8 content tables + `site_text` key/value table for editable fixed section copy), `submissions.ts` (`form_submissions` table with `type`/`status` CHECK constraints), `staffUsers.ts` (admin accounts).
- Seed: `scripts/src/seed.ts` — idempotent admin account + seeds content tables from App.tsx data when empty (gallery left empty).
- Object storage helpers: `artifacts/api-server/src/lib/objectStorage.ts`.

## Architecture decisions

- Content CRUD is generated by a single `crudRouter` factory in `routes/content.ts` (list public; create/update/delete/reorder behind `requireAuth`), so all 8 content types share one code path.
- Auth uses `express-session` + `connect-pg-simple` (Postgres-backed sessions) with scrypt password hashing; cookies are `sameSite:none; secure` and the app trusts the proxy so the admin works cross-origin through the shared proxy.
- Public site degrades gracefully: each section maps API data to its view shape and falls back to the original static arrays on empty/error, so the site never renders blank even if the API is down. During load it shows static content, then swaps to DB content.
- The DB does not store programme/video images. The public site keeps the static images as a lookup (programmes by title, videos by `youtubeId`, with a YouTube-thumbnail fallback) so DB-driven items still get artwork.
- Gallery is intentionally not seeded — it returns `[]` until staff upload, which triggers the static-gallery fallback.
- Editable section copy (`site_text`) is a server-canonical key/value model, not a reorderable list: the canonical block list (key, label, multiline, default) lives only in `routes/site-text.ts`. GET merges defaults with DB overrides (so the admin always sees the full field set and the public site has defaults even with an empty table); PATCH upserts by key and 404s on unknown keys. No seed dependency.

## Product

Grace High School (Gayaza, Uganda) marketing website. Sections: about, programmes, achievements, news, updates, **resources** (downloadable past papers & holiday work), campus, videos, admissions, contact. The resources section lets students download files the school shares.

A staff-only **Admin Dashboard** (preview `/admin/`) lets staff manage all editable content — news ticker, gallery (with image upload), testimonials, videos, programmes, stats, values, admission steps, and resources — with create/edit/delete, reorder, and show/hide. A **Section Text** tab also lets staff edit the fixed section copy (About heading/body/mission, Programmes heading/intro, Admissions heading/intro). Access is gated by a username+password login (server-side Postgres sessions). The public site reflects these changes live via public GET APIs.

## Recent changes

- **Admin upload/auth fix — shared client now sends `credentials: "include"`.** Gallery photo upload (and any authed admin call) could 401 on `POST /api/storage/uploads/request-url` because the shared `customFetch` (`lib/api-client-react/src/custom-fetch.ts`) used the fetch default `credentials: "same-origin"`, which drops the `SameSite=None; Secure` session cookie in the cross-site admin iframe. The server was already correct (CORS `credentials:true`, cookie `SameSite=None; Secure`); the client was the missing half. Verified the full server flow end-to-end over HTTPS (login → `/auth/me` → `request-url` → GCS PUT all 200). Note: `credentials:"include"` can't override a browser that hard-blocks third-party cookies — then the admin must be opened in a first-party tab.

- **"Updates for Parents" noticeboard → dedicated `/updates` page.** The long parent-noticeboard list was moved off the home page to its own clickable detail page (`/updates`) so the home stays short. The 5 notices were lifted to App-scope `updateItems` with a reusable `updateCard(u, i)` render fn and an `updatesBody` JSX (full list + "Never miss an update" contact card). `renderDetailPage()` gained an `isUpdates` branch (`useRoute("/updates")`) that renders `updatesBody` via the shared `detailLayout()`. The home `#updates` section is kept (so the nav "Updates" item still scrolls to it) but slimmed to a compact teaser: the 2 most recent notices (`updateItems.slice(0,2).map(updateCard)`) + a centered "View All N Updates" button calling `openDetail("/updates")` (so scroll-restore/back behaves like the other detail pages). `updateItems` are static (not API-driven), matching the prior inline behavior.

- **3 popups → dedicated pages (wouter routing).** The Vision & Mission, Programme-subjects, and Admissions-topic click-to-open popups are now separate pages with their own URLs inside the same SPA: `/about/vision-mission`, `/programmes/:slug`, `/admissions/:slug` (slug = `slugify(title)`). They share the header/footer chrome, are NOT in the main nav, and are reached only by clicking the existing buttons/tiles (pattern copied from Gayaza High School, but with Grace's green+gold palette — no red). The old `modal` state + Escape effect + INFO POPUP render block were removed; the three popup bodies were lifted to App scope as reusable values/fns (`visionMissionBody`, `programmeBody(p)`, `admissionSlides[]`). Home sections render under `{isHome && …}`; `renderDetailPage()` renders the matched detail page via `detailLayout()` (a plain function — GREEN_DARK hero band w/ `padding-top:170px` to clear the fixed logo overhang, clickable breadcrumb, title/subtitle, content area, Back-to-Home), with a graceful not-found fallback for unknown slugs. `scrollTo`/`goHome` were made cross-route aware: on a detail page they stash a `pendingScroll` ref then `navigate("/")`, and a `useEffect([isHome])` (double-rAF) scrolls to the target after the home page remounts; the `.reveal` IntersectionObserver dep is `[isHome]` so remounted home sections get re-observed. Nav stays solid on detail pages via `scrolled = scrolledY || !isHome`.

- **Form submissions ("Enquire / Apply Now" → admin).** The public enquiry form now POSTs to `POST /api/submissions` (public) and stores rows in the `form_submissions` table. Staff view, mark read/new, delete, and **export CSV** from a new **"Enquiries" tab** in the Admin Dashboard (`SubmissionsSection.tsx`); GET/PATCH/DELETE are behind `requireAuth`. `type` (`enquiry`/`application`) and `status` (`new`/`read`) are enforced as enums end-to-end (OpenAPI → generated Zod → route validation → DB CHECK constraints). The public POST is abuse-hardened: in-memory per-IP rate limit (5/min → 429), a hidden `website` honeypot field (silently accepted, never stored), and `maxLength` guards on all free-text fields. Staff can also upload downloadable **S1/S5 application forms** via the new `application_form` resource category (renders as an "Application Forms" group on the public site alongside past papers / holiday work).

- **Above-the-fold restyled to mirror St. Joseph of Nazareth HS (jonahs.ac.ug) layout**, adapted to Grace's green+gold branding (no red). Hero now uses an italic eyebrow ("Be Part of Our Story") + a heavy Inter-800 rotating headline `[word] to Create the Future.` (word cycles from the `HERO_WORDS` const in sync with `heroSlide`) + a single green "Enroll Now" CTA and an outline "Read Our Story" (the old serif "Run With a Vision" h1, `HERO_TAGLINES`, and 3 gold buttons were removed). ## Gotchas

- **`pnpm --filter @workspace/db run push` is interactive-only here** — drizzle-kit prompts for confirmation on a TTY that isn't available to the agent, so it hangs. For schema changes, apply the equivalent raw DDL via `executeSql` (CREATE TABLE / ALTER TABLE … ADD CONSTRAINT) to match the Drizzle schema, then keep the schema file as the source of truth.
- **After adding/removing a route file, restart the API Server workflow** — new Express routes aren't picked up by hot reload; the route 404s until `artifacts/api-server: API Server` is restarted.
